There's a great post by Christian Ekrem called [*Effect Without Effect-TS: Algebraic Thinking in Plain TypeScript*](https://cekrem.github.io/posts/effect-without-effect-ts/). It does something valuable: it takes the core ideas behind Effect (typed errors, explicit dependencies, and composition) and shows you can apply them with plain TypeScript. No framework required. The Result type, the dependency injection pattern, the honest function signatures. It's all good thinking, and if you haven't read it, you should.

This page takes those same principles and shows how awaitly builds on them. The post is honest about the trade-offs of the manual approach: composition past four or five steps gets verbose, error unions spread across module boundaries, and structured concurrency is a separate problem. It also leaves one hard problem unaddressed: **unexpected errors** — the throws you didn't plan for.

awaitly picks up where the manual approach gets repetitive, and adds a safety net for the errors you didn't anticipate. It does introduce its own ceremony: `ok/err`, `run`, `step`, `step.try`, `ErrorsOf`, and `UnexpectedError` are all concepts you have to learn. The claim is not that awaitly has no ceremony, but that it is **fixed and reusable**, whereas the manual approach recreates it in every module.

For most code, `run()` is the starting point. `createWorkflow()` is there when you want orchestration features like retries, caching, observability, or managed dependency wiring.

A fair framing of the three approaches:

```text
Raw async/await       Manual Result pattern      awaitly
──────────────────    ──────────────────────      ──────────────────────
Least setup           Most boilerplate            Moderate setup
Least honesty         Most explicit control       Lower repeated boilerplate
Errors invisible      Errors visible (expected)   Stronger guarantees
                      Unexpected errors: 🤷        Unexpected errors: typed
```

## The function that lies to you

The post starts with this function. We've all written it:

```typescript
async function signupUser(email: string, password: string): Promise<User> {
  if (!isValidEmail(email)) throw new Error("Invalid email");

  const existing = await db.findUserByEmail(email);
  if (existing) throw new Error("Email already registered");

  const user = await db.createUser({
    email,
    passwordHash: await hash(password),
  });

  await emailService.sendWelcome(user.email);
  await analytics.track("user_signed_up", { userId: user.id });

  return user;
}
```

The return type says `Promise<User>`. That's a lie. The function can fail in at least four ways, calls a database, sends an email, tracks analytics. None of that is in the signature.

The post's solution is solid: return errors as values, inject dependencies, compose with `andThen`. These are good principles worth learning regardless of what tools you use. The question is not whether there is ceremony, but where it lives.

## Idea 1: Honest errors

The manual approach defines a union and a Result type:

```typescript
type SignupError =
  | { _tag: "InvalidEmail" }
  | { _tag: "EmailTaken"; email: string }
  | { _tag: "DbError"; cause: unknown }
  | { _tag: "EmailServiceDown" };

type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };
```

Then the function returns `Promise<Result<User, SignupError>>` and the caller switches on `error._tag` with an `assertNever` guard.

This works well. A disciplined team could also factor out the `Result` type and `assertNever` into a shared utility file, and use helper constructors for each error variant. That's a fair point; the manual approach doesn't have to be maximally verbose.

But even with shared helpers, each module still owns its error union. Add a step, update the union. Remove a step, update the union. Across a codebase, that's real maintenance.

### The awaitly equivalent

```typescript
const validateEmail = async (
  email: string

  isValidEmail(email) ? ok(email) : err("INVALID_EMAIL");

const checkDuplicate = async (
  email: string

  const existing = await db.findUserByEmail(email);
  return existing ? err("EMAIL_TAKEN") : ok();

```

  <TabItem label="Manual (with helpers)">
```typescript
// shared/result.ts (you write this once)
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };
const ok = <T>(value: T): Result<T, never> => ({ ok: true, value });
const err = <E>(error: E): Result<never, E> => ({ ok: false, error });

function assertNever(x: never): never {
  throw new Error(`Unexpected: ${JSON.stringify(x)}`);
}

// signup.ts
const validateEmail = async (
  email: string
): Promise<Result<string, { _tag: "InvalidEmail" }>> =>
  isValidEmail(email)
    ? ok(email)
    : err({ _tag: "InvalidEmail" });

const checkDuplicate = async (
  email: string
): Promise<Result<void, { _tag: "EmailTaken"; email: string }>> => {
  const existing = await db.findUserByEmail(email);
  return existing
    ? err({ _tag: "EmailTaken", email })
    : ok(undefined);
};
```
  </TabItem>
</Tabs>

With shared helpers, the manual approach is already pretty close. The main differences: `AsyncResult<T, E>` as a type alias, string literal error codes instead of `{ _tag }` objects, and `ok()` handling the void case (no `ok(undefined)`). Small wins, not transformative on their own.

## Idea 2: Honest dependencies

The post's second idea is to put dependencies in the function signature. We agree completely; this is the simplest, most underrated pattern in the post. awaitly leans into the same idea with two different shapes.

### Shape 1: `run` + `ErrorsOf` - plain functions, close to the shape of the blog post

This is the lighter approach. Your functions stay plain, you use the `fn(args, deps)` pattern from the blog post directly, and `run()` provides the execution boundary while `step()` unwraps results and short-circuits on failure:

```typescript
import { ok, err, type AsyncResult, type ErrorsOf } from 'awaitly';
import { run } from 'awaitly/run';

// Plain functions, with no framework registration
const validateEmail = async (email: string): AsyncResult<string, "INVALID_EMAIL"> =>
  isValidEmail(email) ? ok(email) : err("INVALID_EMAIL");

const findUser = async (email: string): AsyncResult<User | null, "DB_ERROR"> =>
  ok(await db.findUserByEmail(email));

const checkNotTaken = async (user: User | null): AsyncResult<void, "EMAIL_TAKEN"> =>
  user ? err("EMAIL_TAKEN") : ok();

const createUser = async (input: CreateUserInput): AsyncResult<User, "DB_ERROR"> =>
  ok(await db.createUser(input));

const sendWelcome = async (email: string): AsyncResult<void, "EMAIL_SERVICE_DOWN"> =>
  ok(await emailService.sendWelcome(email));

// Group deps and derive the error union: no hand-maintained type
const deps = { validateEmail, findUser, checkNotTaken, createUser, sendWelcome };
type SignupError = ErrorsOf<typeof deps>;
// = "INVALID_EMAIL" | "DB_ERROR" | "EMAIL_TAKEN" | "EMAIL_SERVICE_DOWN"

// run() provides the execution boundary; step() unwraps results and short-circuits on failure
const result = await run<User, SignupError>(async ({ step }) => {
  const email = await step("validate", validateEmail(rawEmail));
  const existing = await step("find", findUser(email));
  await step("checkNotTaken", checkNotTaken(existing));
  const user = await step("create", createUser({ email, passwordHash: await hash(password) }));
  await step("welcome", sendWelcome(email));
  return user;
});
```

You can call functions directly, or keep the explicit `fn(args, deps)` style from the blog post and add `run()` as the error boundary:

```typescript
// Define the deps type once, derive errors from it
type SignupDeps = typeof deps;
type SignupErrors = ErrorsOf<SignupDeps>;

// Core function, testable with any deps
const signupUser = async (
  args: { email: string; password: string },
  deps: SignupDeps,
) => {
  return run<User, SignupErrors>(async ({ step }) => {
    const email = await step("validate", deps.validateEmail(args.email));
    const existing = await step("find", deps.findUser(email));
    await step("checkNotTaken", deps.checkNotTaken(existing));
    const user = await step("create", deps.createUser({ email, passwordHash: await hash(args.password) }));
    await step("welcome", deps.sendWelcome(email));
    return user;
  });
};

// Wire once at the edge
const signup = (args: { email: string; password: string }) =>
  signupUser(args, deps);

// Call site is clean
const result = await signup({ email: rawEmail, password });

// Tests inject fakes directly, no mocking library
const testResult = await signupUser(
  { email: "test@example.com", password: "pass123" },
  {
    validateEmail: async (e) => ok(e),
    findUser: async () => ok(null),
    checkNotTaken: async () => ok(),
    createUser: async () => ok({ id: "1", email: "test@example.com" }),
    sendWelcome: async () => ok(),
  }
);
```

This keeps the core function maximally testable (inject real or mock deps), and `run()` provides the error boundary while `step()` handles composition. The test story is identical to the blog post: pass fake deps. No mocking library.

<Aside type="tip" title="Optional wiring helper">
awaitly also ships a `bindDeps` helper for currying `fn(args, deps)` functions once at the edge, but the core pattern is just plain `fn(args, deps)` with a closure at the composition boundary, as shown above.
</Aside>

### Shape 2: `createWorkflow` for when you need more

Let's be honest about what this is: `createWorkflow` is a framework-shaped pattern. You're registering functions with a coordinator, and it gives you back a runner. Some people will see that as structure; others will see it as a framework they didn't ask for. That's a fair reaction.

The payoff is that the error union is inferred automatically from the registered functions, with no separate `ErrorsOf<...>` type needed. You also get step caching, retry/timeout config, and observability events:

```typescript
import { createWorkflow, ok, err, type AsyncResult } from 'awaitly';

// Declare deps, error union computed automatically
const signup = createWorkflow("signup", {
  validateEmail,
  findUser,
  checkNotTaken,
  createUser,
  sendWelcome,
});

const result = await signup.run(async ({ step, deps }) => {
  const email = await step("validate", deps.validateEmail(rawEmail));
  const existing = await step("find", deps.findUser(email));
  await step("checkNotTaken", deps.checkNotTaken(existing));
  const user = await step("create", deps.createUser({ email, passwordHash: await hash(password) }));
  await step("welcome", deps.sendWelcome(email));
  return user;
});

// result.error is:
// "INVALID_EMAIL" | "DB_ERROR" | "EMAIL_TAKEN" | "EMAIL_SERVICE_DOWN" | UnexpectedError
// ^ all inferred from deps                                               ^ see below
```

**In most cases, start with `run`.** Reach for `createWorkflow` when you want orchestration features. Both catch unexpected errors.

### Compared to the manual approach

The blog post's dependency pattern looks like this:

```typescript
type SignupDeps = {
  readonly findUserByEmail: (email: string) => Promise<Result<User | null, DbError>>;
  readonly createUser: (input: CreateUserInput) => Promise<Result<User, DbError>>;
  readonly sendWelcomeEmail: (email: string) => Promise<Result<void, EmailError>>;
  readonly trackEvent: (name: string, props: Record<string, string>) => Promise<void>;
};

async function signupUser(
  deps: SignupDeps,
  email: string,
  password: string,
): Promise<Result<User, SignupError>> {
  const parsed = parseEmail(email);
  if (!parsed.ok) return { ok: false, error: { _tag: "InvalidEmail" } };

  const existing = await deps.findUserByEmail(parsed.value);
  if (!existing.ok)
    return { ok: false, error: { _tag: "DbError", cause: existing.error } };
  if (existing.value)
    return { ok: false, error: { _tag: "EmailTaken", email } };

  const created = await deps.createUser({
    email: parsed.value,
    passwordHash: await hash(password),
  });
  if (!created.ok)
    return { ok: false, error: { _tag: "DbError", cause: created.error } };

  // ...
  return { ok: true, value: created.value };
}
```

This is also clean, explicit code. A skilled TypeScript developer could factor out the early-return checks into a helper and shrink this further. The difference is:

1. **`step()` absorbs the manual early-exit checks.** You stop writing `if (!r.ok) return r` after every call.
2. **`ErrorsOf<typeof deps>` infers the union.** You don't hand-maintain `SignupError = ... | ... | ...`.
3. **The test story is the same.** Both pass fake deps. Neither needs a mocking library.

## Idea 3: Composition

The post introduces `andThen`, a flatMap for async results, and is refreshingly honest about where it runs out of steam: "Past four or five steps, the nesting gets painful." It points to Effect-TS generators as the real solution, and says "I'm not going to build that in this post."

That honesty is part of what makes the post good. And it's the exact gap `step()` is designed to fill, taking the same composition idea one step further without generators:

```typescript
  const validated = await validateEmail(email);
  if (!validated.ok) return validated;

  const existing = await findUser(validated.value);
  if (!existing.ok) return existing;

  const checked = await checkNotTaken(existing.value);
  if (!checked.ok) return checked;

  const account = await createAccount(validated.value);

  if (!welcomed.ok) return welcomed;

  return account;
}
```

The manual early-return version (third tab) is actually pretty readable. For simple linear flows it's fine. The cost shows up when error types need to compose across the returns; each one potentially widens the union, and you're responsible for making sure the function's return type covers all of them.

## Taking it further: unexpected errors

The post does a great job covering **expected errors**: validation failures, duplicate emails, and database errors you planned for. That alone is a real improvement over raw try/catch, and worth adopting even without any library.

One gap the post leaves open is what happens when something you *didn't* plan for throws.

In real code, this happens constantly:

- A dependency throws a `TypeError` because someone passed `undefined`
- A JSON parse fails inside a library you don't control
- A network call throws `ECONNREFUSED` instead of returning a clean error
- A third-party SDK throws something that isn't even an `Error`

The manual approach, as usually presented, does not include a built-in answer for this. You could wrap every call in try/catch and map unknown throws to a catch-all error variant, but that reintroduces some of the ceremony the Result pattern tries to reduce. In many codebases, teams skip that extra boundary, so `Result<T, SignupError>` covers only the errors they modeled and unexpected throws remain outside the type.

This is the same kind of incompleteness the post starts by highlighting, now on the error side instead of the success side. `Promise<User>` hid failure modes. `Result<User, SignupError>` improves that picture significantly, but it can still look complete on the type level even when unexpected runtime failures are possible.

### awaitly's answer: `UnexpectedError`

Every `run()` and `createWorkflow()` boundary catches unexpected throws and wraps them in `UnexpectedError`:

```typescript
const result = await run(async ({ step }) => {
  const email = await step("validate", validateEmail(rawEmail));
  const existing = await step("find", findUser(email));
  await step("checkNotTaken", checkNotTaken(existing));

  // Imagine createAccount throws a TypeError internally
  const account = await step("create", createAccount(email));

  return account;
});

// result.error is:
// "INVALID_EMAIL" | "EMAIL_TAKEN" | "DB_ERROR" | UnexpectedError
//                                                 ^^^^^^^^^^^^^^
//                                    caught, wrapped, in the union
```

The `UnexpectedError` appears **in the type**. TypeScript nudges you to handle it:

```typescript
import { isUnexpectedError } from 'awaitly';

if (!result.ok) {
  if (isUnexpectedError(result.error)) {
    // Infrastructure failure: log the cause, return 500
    logger.error('Unexpected failure', { cause: result.error.cause });
    return res.status(500).json({ error: 'Internal error' });
  }

  // Now TypeScript narrows to your domain errors
  switch (result.error) {
    case "INVALID_EMAIL":
      return res.status(400).json({ error: 'Bad email' });
    case "EMAIL_TAKEN":
      return res.status(409).json({ error: 'Already registered' });
    case "DB_ERROR":
      return res.status(500).json({ error: 'Try again later' });
  }
}
```

If you want a fully closed error union with no `UnexpectedError` class, use `catchUnexpected` to map it into your domain:

```typescript
const result = await run(
  async ({ step }) => {
    // ... workflow body
  },
  {
    catchUnexpected: (cause) => "INTERNAL_ERROR" as const,
  }
);
// result.error: "INVALID_EMAIL" | "EMAIL_TAKEN" | "DB_ERROR" | "INTERNAL_ERROR"
// ^ fully closed union, no UnexpectedError
```

<Aside type="note" title="Could you build this manually?">
Yes. You could write a `runSafe` wrapper that catches unknown throws and maps them to a tagged error. That's maybe 15 lines of code. The point isn't that it's hard; it's that this extra boundary is often left as an exercise for application code. awaitly includes it by default.
</Aside>

### `step.try()` for throwing code you don't control

When calling code that throws (third-party SDKs, JSON.parse, DOM APIs), `step.try()` catches the throw and converts it to a typed error:

```typescript
const result = await run(async ({ step }) => {
  // Static error code
  const data = await step.try(
    "parse",
    () => JSON.parse(rawInput),
    { error: "PARSE_FAILED" as const }
  );

  // Dynamic error mapping
  const response = await step.try(
    "callSDK",
    () => thirdPartySDK.call(data),
    { onError: (e) => ({
        type: "SDK_FAILED" as const,
        message: e instanceof Error ? e.message : String(e),
      })
    }
  );

  return response;
});
// result.error: "PARSE_FAILED" | { type: "SDK_FAILED"; message: string } | UnexpectedError
```

This is another concept to learn (`step.try` vs `step`). That's real cognitive overhead. The trade-off is that it replaces the manual try/catch-and-map-to-Result pattern you'd otherwise write around every throwing API.

## The trade-off scorecard

Here's what each approach requires you to write, learn, and maintain:

| Dimension | Raw async/await | Manual Result | awaitly |
|---|---|---|---|
| Setup cost | None | `Result` type, `ok`/`err` helpers, `assertNever` | `npm install`, learn `run`/`step`/`ErrorsOf` |
| Per-module cost | Low | Error union + early returns per call | `step()` calls |
| Error visibility | None (catch unknown) | Expected errors typed | Expected + unexpected typed |
| Error union maintenance | N/A | Hand-maintained | Inferred (`ErrorsOf<typeof deps>` or `createWorkflow`) |
| Unexpected errors | Uncaught rejections | Usually uncaught unless you add a safe boundary | `UnexpectedError` in the union |
| Composition at scale | try/catch nesting | `andThen` nesting or early returns | `step()` |
| Dep injection | Module imports | `fn(deps, ...args)` | `fn(args, deps)` with plain wiring, or `createWorkflow` |
| Test story | Often mocks or module stubs | Pass fake deps | Pass fake deps |
| Concepts to learn | 0 | ~3 (Result, unions, assertNever) | ~5 (ok/err, run, step, ErrorsOf, UnexpectedError) |

awaitly does introduce its own ceremony: result constructors, run boundaries, and step-based composition. The claim is that this ceremony is fixed (learn once) and centralized (in the library), whereas the manual approach distributes it across every module that handles errors.

## When each approach fits

**Raw async/await** is fine for scripts, small services, and code where error handling is genuinely not critical. No shame in it.

**Manual Result patterns** work well when you have a small codebase, a disciplined team, and you want maximum control with zero dependencies. The blog post's patterns are sound. If you're comfortable maintaining error unions by hand and you've already factored out your own helpers, you may not need anything else.

**awaitly** starts earning its keep when:
- Error unions span multiple modules and manual maintenance becomes a drag
- You want unexpected errors in the type system, not silently dropped
- You need retry, timeout, circuit breaker, or saga on top of the same pattern
- You're composing more than four or five steps regularly
- You want observability (step events, OpenTelemetry) without instrumenting every function

## The takeaway

The blog post teaches the right ideas. Typed errors, explicit dependencies, and honest function signatures make code better regardless of what tools you use. If you read it and start applying those patterns in plain TypeScript, you're already ahead of most codebases.

awaitly takes those same principles one step further: it infers the error union with `ErrorsOf`, removes the repeated early-exit checks with `step()`, and catches the unexpected throws that manual Result types usually leave unhandled. It helps you keep the direct async/await style people already like, while making errors and failures more honest.

The ideas are similar. The difference is where the ceremony lives: hand-written in every module, or centralized behind a small execution boundary.
